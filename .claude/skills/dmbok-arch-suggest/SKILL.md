# DMBOK アーキテクチャ提案

## 概要

dmbok-assess の出力（アセスメント結果）を入力として、`references/` 配下のソリューション設計方針・パターンに沿った具体的なアーキテクチャ改善提案を行う。

## 実行フロー

### Step 1: リファレンスの読み込み

以下を読み込んで提案の基準とする:

- `references/solution-design-policy.md` — 設計の基本方針・原則、技術スタック選定指針、ネットワーク設計方針、セキュリティ・ガバナンス方針
- `references/solution-pattern.md` — 提供可能なアーキテクチャパターン（構成要素、データフロー、長所/短所、選定基準）

### Step 2: アセスメント結果の読み込み

対象のアセスメント結果を読み込む。以下の優先順で探索する:

1. ユーザーが引数でパスを指定した場合はそのファイル
2. `output/assessment_*.json` の最新ファイル
3. `output/assessment_*.md` の最新ファイル

読み込む情報:
- 各領域のスコア・課題（issues）・改善提案（recommendations）
- ロードマップ（短期・中期・長期）
- 評価の制約・留意事項

### Step 3: 課題の分析と優先度付け

アセスメント結果の課題を以下の観点で分類・優先度付けする:

1. **アーキテクチャで解決可能な課題** — ツール導入、サービス追加、構成変更で対応できるもの
2. **プロセス・組織で解決すべき課題** — ガバナンス体制、教育、ポリシー策定などアーキテクチャだけでは解決できないもの
3. **情報不足で判断保留の課題** — 追加ヒアリングが必要なもの

アーキテクチャで解決可能な課題を中心に提案を組み立てる。プロセス・組織系の課題についてはアーキテクチャ面でのサポート（例: ガバナンスツールの導入）を提案する。

### Step 4: アーキテクチャ提案の作成

リファレンスの設計方針・パターンに準拠し、以下の構造で提案を作成する:

#### 4.1 提案の原則

提案は必ず `solution-design-policy.md` の以下の方針に沿うこと:

- **クラウドファースト**: AWS/GCP 中心、サーバレス優先
- **再利用性・再現性の最大化**: IaC 前提、パターン化
- **運用性の重視**: 監視・保守の容易さ
- **セキュリティデザイン**: 設計段階からセキュリティを考慮
- **コスト最適化**: 長期運用を考慮したリソース選択

#### 4.2 パターンマッチング

現行アーキテクチャと課題を `solution-pattern.md` のパターン選定基準と照合し、以下を判定する:

- 現行アーキテクチャに最も近いパターンはどれか
- 課題解決に適したパターンへの移行・拡張は可能か
- パターンの長所/短所が顧客の状況に合致するか

#### 4.3 提案の構成

各提案は以下の項目を含めること:

```
## 提案 N: [提案タイトル]

### 対象課題
- アセスメントで検出された課題の要約

### 対象 DMBOK 領域
- 関連する DMBOK 領域名

### 提案内容
- 具体的なアーキテクチャ変更・追加の説明
- 採用する AWS/GCP サービスや SaaS の明示

### アーキテクチャ構成
- 主要コンポーネントと役割
- データフロー（テキストベースまたは図で説明）

### 適用パターン
- solution-pattern.md のどのパターンに該当するか（該当なしの場合はその旨記載）
- パターンからのカスタマイズ点

### 期待効果
- スコア改善の見込み（例: メタデータ管理 2→3）
- 定性的な効果

### 導入コスト・期間の目安
- 概算の導入期間
- 主要なコスト要素

### リスク・留意事項
- 導入時のリスクと対策
```

### Step 5: ロードマップとの統合

アセスメントのロードマップ（短期・中期・長期）と提案を統合し、実行順序を整理する:

- 短期（0〜3ヶ月）: クイックウィン。既存構成への追加で実現可能な改善
- 中期（3〜6ヶ月）: 新サービス導入やアーキテクチャ拡張
- 長期（6〜12ヶ月）: 大規模なアーキテクチャ変更や移行

### Step 6: レポート出力

`output/arch_suggestion_YYYYMMDD.md` に Markdown 形式で提案レポートを出力する。

レポート構成:
1. エグゼクティブサマリー — 提案全体の概要（1段落）
2. アセスメント結果の要約 — 入力となった課題の要約
3. 提案一覧 — 提案のサマリーテーブル（対象領域、提案名、優先度、期間）
4. 各提案の詳細 — Step 4.3 の構成に従う
5. 統合ロードマップ — 時系列での実行計画
6. 前提条件・制約事項

## 注意事項

- レポートは日本語で記述する
- 提案は `references/` のソリューション設計方針・パターンに準拠すること。方針に記載のないサービスや構成を提案する場合は、その理由を明記する
- アセスメントで「情報不足」とされた領域については、追加ヒアリング項目を提案に含める
- コスト見積もりは概算レベルとし、詳細見積もりには別途調査が必要な旨を記載する
- 現行アーキテクチャからの移行パスを考慮し、ビッグバン的な変更は避ける
- 組織・プロセス面の課題についても、アーキテクチャで支援できる部分は提案に含める
