# 03. データモデリングとデザイン (Data Modeling & Design)

## 概要

データモデリングとデザインは、データ要件を発見・分析・表現し、概念・論理・物理の各レベルでデータモデルを設計するプロセスである。ビジネス要件をデータ構造に落とし込む橋渡しの役割を果たす。

## 評価観点

| # | チェック項目 | 確認内容 |
|---|-------------|----------|
| 1 | 概念データモデルの存在 | ビジネスエンティティとその関係を示す概念モデルがあるか |
| 2 | 論理データモデルの存在 | 正規化された論理モデルが設計・文書化されているか |
| 3 | 物理データモデル（DDL） | テーブル定義・制約・インデックスが適切に設計されているか |
| 4 | モデリング標準 | 命名規則、データ型標準、正規化ルールが定義されているか |
| 5 | モデルのバージョン管理 | DDL / マイグレーションファイルがバージョン管理されているか |
| 6 | ER図の整備 | ER図が最新の状態で維持されているか |
| 7 | リレーションシップの定義 | 外部キー制約やリレーションが適切に定義されているか |
| 8 | データ型の適切性 | カラムのデータ型が適切に選択されているか |

## 成熟度レベル定義

| レベル | 状態 |
|--------|------|
| 1 - 初期 | データモデルが存在せず、テーブル設計が場当たり的 |
| 2 - 反復可能 | 一部のテーブルにDDLがあるが、概念・論理モデルがない |
| 3 - 定義済み | 概念・論理・物理モデルが文書化され、命名規則が存在する |
| 4 - 管理済み | モデル変更がレビュープロセスを経て、影響分析が行われている |
| 5 - 最適化 | モデルが自動検証され、品質メトリクスで管理されている |

## Git/Backlogで確認すべきポイント

### Gitリポジトリ
- DDLファイル（`*.sql`, `schema/`, `migrations/`）
- ER図（`.erd`, `.puml`, Mermaid記法のファイル）
- ORMモデル定義（`models.py`, `entities/` 等）
- マイグレーションツールの使用（Flyway, Alembic, Liquibase等）
- 命名規則の一貫性（snake_case / camelCase の統一など）

### Backlog
- Documents/Wiki のデータ関連ドキュメント
- DB設計に関するチケット・レビュー記録
- モデル変更の承認フロー
- ER図やデータ辞書のWiki

## よくある課題と改善策

| 課題 | 改善策 |
|------|--------|
| DDLが管理されていない | マイグレーションツールの導入（Alembic等） |
| ER図がない / 古い | DDLからの自動生成、CI連携 |
| 命名規則がバラバラ | Linterによる自動チェック、命名規則ドキュメントの策定 |
| 正規化が不十分 | 定期的なモデルレビュー会の実施 |
