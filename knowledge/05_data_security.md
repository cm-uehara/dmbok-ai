# 05. データセキュリティ (Data Security)

## 概要

データセキュリティは、データの機密性（Confidentiality）、完全性（Integrity）、可用性（Availability）を保護するための方針・手順・技術的措置を管理する領域である。認証・認可・暗号化・監査などを通じてデータ資産を不正アクセスや漏洩から守る。

## 評価観点

| # | チェック項目 | 確認内容 |
|---|-------------|----------|
| 1 | アクセス制御 | データへのアクセス権限が最小権限の原則で管理されているか |
| 2 | 認証・認可の仕組み | 適切な認証方式（MFA等）と認可モデル（RBAC等）が実装されているか |
| 3 | データ暗号化 | 保管時（at rest）と通信時（in transit）の暗号化が適用されているか |
| 4 | 機密データの分類 | データの機密レベルが定義・分類されているか |
| 5 | 監査ログ | データアクセスの監査ログが取得・保管されているか |
| 6 | 秘密情報の管理 | パスワード・APIキー等がコードにハードコードされていないか |
| 7 | 脆弱性対策 | SQLインジェクション等の脆弱性対策が実装されているか |
| 8 | データマスキング | テスト環境等での機密データのマスキング方針があるか |

## 成熟度レベル定義

| レベル | 状態 |
|--------|------|
| 1 - 初期 | セキュリティ対策が最小限。秘密情報がコードに含まれる場合がある |
| 2 - 反復可能 | 基本的なアクセス制御はあるが、体系的な管理がされていない |
| 3 - 定義済み | セキュリティポリシーが文書化され、暗号化・アクセス制御が標準化されている |
| 4 - 管理済み | セキュリティ監査が定期的に実施され、脆弱性管理が定量化されている |
| 5 - 最適化 | セキュリティが DevSecOps として開発プロセスに統合されている |

## Git/Backlogで確認すべきポイント

### Gitリポジトリ
- `.gitignore` での機密ファイル除外設定
- `.env.example` の存在（`.env` がコミットされていないか）
- シークレット管理（Vault, AWS Secrets Manager等の利用）
- 認証・認可の実装コード
- SQLパラメータバインディングの使用
- HTTPS / TLS設定
- セキュリティ系のCI設定（SAST, dependency check等）
- `SECURITY.md` の存在

### Backlog
- セキュリティインシデントの管理チケット
- セキュリティレビューの記録
- アクセス権限の管理チケット

## よくある課題と改善策

| 課題 | 改善策 |
|------|--------|
| 秘密情報がコードに含まれる | git-secrets / gitleaks の導入、Secret Manager の利用 |
| アクセス権限が過剰 | 定期的なアクセス権レビュー、最小権限の原則の適用 |
| 監査ログがない | アクセスログの体系的な取得・保管の仕組み構築 |
| セキュリティテストが不十分 | CI/CDにセキュリティスキャンを統合 |
