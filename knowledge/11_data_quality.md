# 11. データ品質 (Data Quality)

## 概要

データ品質は、データが使用目的に対して適切であることを保証するための計画・実装・管理を行う領域である。正確性、完全性、一貫性、適時性、一意性、妥当性の6つの品質次元を中心に、データの信頼性を評価・改善する。

## 品質の6次元

| 次元 | 定義 |
|------|------|
| 正確性 (Accuracy) | データが現実世界の事実を正しく反映しているか |
| 完全性 (Completeness) | 必要なデータが欠損なく存在するか |
| 一貫性 (Consistency) | 同じデータが異なる場所で矛盾なく一致しているか |
| 適時性 (Timeliness) | データが必要なタイミングで利用可能か |
| 一意性 (Uniqueness) | 同じエンティティに対するデータの重複がないか |
| 妥当性 (Validity) | データが定義されたルール・制約に適合しているか |

## 評価観点

| # | チェック項目 | 確認内容 |
|---|-------------|----------|
| 1 | データ品質ルールの定義 | NULL制約、値範囲、フォーマット等のルールが定義されているか |
| 2 | バリデーション実装 | 入力値の検証がアプリケーション層で実装されているか |
| 3 | DB制約の活用 | NOT NULL, UNIQUE, CHECK, FK等のDB制約が適切に設定されているか |
| 4 | データ品質モニタリング | データ品質を定期的に測定・監視する仕組みがあるか |
| 5 | 異常検知 | データ異常（欠損、外れ値、重複）を検知する仕組みがあるか |
| 6 | データクレンジング | データの修正・整備プロセスが定義されているか |
| 7 | 品質メトリクス | データ品質のKPI・SLAが定義されているか |
| 8 | テストの充実度 | データに関するテスト（ユニットテスト、統合テスト）が整備されているか |

## 成熟度レベル定義

| レベル | 状態 |
|--------|------|
| 1 - 初期 | データ品質の概念がなく、問題が起きてから対応する |
| 2 - 反復可能 | 基本的なバリデーションはあるが、品質の測定・監視がされていない |
| 3 - 定義済み | 品質ルールが文書化され、バリデーション・DB制約が標準化されている |
| 4 - 管理済み | データ品質が定量的に測定され、KPI/SLAとして管理されている |
| 5 - 最適化 | データ品質が自動監視され、異常検知と自動修復の仕組みが機能している |

## よくある課題と改善策

| 課題 | 改善策 |
|------|--------|
| NULL値やデフォルト値が多い | NOT NULL 制約の追加、必須入力の強化 |
| 重複データが発生する | UNIQUE 制約、重複チェック処理の導入 |
| 品質の可視化ができていない | データ品質ダッシュボードの構築 |
| テストが不足 | データ品質テストフレームワーク（Great Expectations等）の導入 |
| データ修正が場当たり的 | クレンジングプロセスの標準化、根本原因分析の実施 |
