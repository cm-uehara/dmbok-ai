# DMBOK コンサルティング（壁打ち・相談）

DMBOKのナレッジベースに基づき、データマネジメントに関する相談に対話的にアドバイスする。

## Step 1: ナレッジベースの参照

相談テーマに該当する領域のナレッジを読み込む:

- `knowledge/00_overview.md` — 全体像の把握
- 該当領域のファイル（例: データガバナンスの相談なら `knowledge/01_data_governance.md`）

複数領域にまたがる場合は関連するファイルをすべて読み込む。

## Step 2: アセスメント結果の活用

`output/` に既存のアセスメントレポートがある場合、相談の出発点として活用する:

1. `output/assessment_*.json` を読み込み、各領域のスコアを把握する
2. 相談テーマに該当する領域のスコア・所見・課題を確認する
3. ロードマップ（short_term / mid_term / long_term）の進捗状況を確認する

アセスメント結果がある場合は「前回の診断ではこの領域がレベルXでした」のように具体的な現在地から話を始められる。ない場合は、ユーザーとの対話で現状を把握する。

## Step 3: アドバイス提供

ナレッジベースの「よくある課題と改善策」セクションと「成熟度レベル定義」を踏まえてアドバイスする。

### 領域間の依存関係を考慮する

11領域は独立ではなく、以下の依存関係がある。改善の順序を提案する際に考慮すること:

```
データガバナンス（中核・ハブ）
 ├── データアーキテクチャ（全体設計の土台）
 │    ├── データモデリングとデザイン
 │    ├── データストレージとオペレーション
 │    └── データ統合と相互運用性
 ├── データセキュリティ（ポリシー策定が前提）
 ├── 参照データとマスターデータ（ガバナンス + アーキテクチャが前提）
 ├── メタデータ管理（他領域の定義情報を管理）
 ├── データ品質（メタデータ管理との連携が効果的）
 ├── データウェアハウスとBI（上流の整備度に依存）
 └── ドキュメントとコンテンツ管理
```

**典型的な改善の順序:**
1. まずガバナンスの枠組み（ポリシー・役割定義・標準）を最低限整える
2. アーキテクチャで全体像を描く
3. その上で個別領域（品質、マスターデータ等）に着手する

ガバナンスなしに個別領域を改善しようとすると、属人的な取り組みに終わりやすい。

### よくあるアンチパターン

相談の中で以下のパターンが見えたら指摘する:

- **ツール先行**: ガバナンスや要件定義なしにツールを導入し、定着しない
- **全社一斉展開**: スモールスタートせず、最初から全社適用を目指して頓挫する
- **文書化偏重**: ドキュメントを大量に作るが運用・改善のサイクルがない（レベル3で停滞）
- **IT部門だけの取り組み**: ビジネス側の巻き込みがなく、実務と乖離する
- **完璧主義**: レベル5を最初から目指し、レベル2→3の地道な改善を軽視する

## 注意事項

- レポートや回答は日本語で記述する
- DMBOK用語を使う際は平易な言葉で補足する（例:「データスチュワード（データの管理責任者）」）
- 「完璧を目指す」ではなく「現在のレベルから1つ上がるために何をするか」に焦点を当てる
